# 八个脆弱点收口报告

更新时间：2026-04-14  
适用子项目：`langchain-miniopenclaw/backend`

## 1. 背景

在前序评测与人工复核中，当前主链虽已具备较强的可用性与领域问答能力，但仍暴露出 8 个典型脆弱点：

1. Deep 生成层脆弱
2. coverage 判定偏启发式
3. files-first 结构摘要仍偏启发式
4. 索引 schema 迁移不成熟
5. route explanation 与真实执行存在轻微脱节
6. 图谱证据虽已升级，但仍需进一步前置为一等证据
7. release gate 还不够严格
8. 运行时可观测性不足

本轮目标不是增加新功能，而是对这些脆弱点做工程化收口。

## 2. 本轮落实的改动

### 2.1 Deep 生成层增强

落点：

- `backend/services/qa_service/runtime_support.py`
- `backend/services/qa_service/models.py`
- `backend/tests/test_qa_service.py`

已完成：

- 为最终 grounded generation 增加可配置重试策略
- Deep 模式生成失败后，不再直接掉入极短 deterministic fallback
- Deep 模式先尝试 `deep -> quick grounded fallback`
- 异常记录改为 `异常类名 + 具体信息`
- 返回 `generation_diagnostics`

解决的问题：

- `planner_deterministic_fallback` 一旦触发即质量塌缩
- `deep_llm_fallback:` 为空，难以定位真实异常

### 2.2 coverage 完整性增强

落点：

- `backend/services/qa_service/evidence_coverage.py`

已完成：

- 新增 `requested_facets`
- 将 query 中显式请求的关键维度映射为 coverage 约束
- 对 `composition / efficacy / indication / comparison / origin / path_reasoning` 增加最小 facet 约束
- coverage summary 中新增 `requested_facets`

解决的问题：

- 过去“证据似乎够了”但问题维度没覆盖完整
- coverage 过于依赖全局信号，不够贴近用户实际提问角度

### 2.3 files-first schema 迁移与版本保护

落点：

- `backend/services/retrieval_service/files_first_support.py`
- `backend/services/retrieval_service/engine.py`
- `backend/tests/test_files_first_support.py`
- `backend/tests/test_retrieval_engine.py`

已完成：

- files-first 增加 schema version
- 新增 `files_first_meta`
- 新增兼容检查：
  - `files_first_schema_version`
  - `files_first_schema_compatible`
  - `files_first_schema_migrated`
- 旧 schema 启动时自动检测
- 优先尝试重建迁移
- 若 Windows 文件锁阻止替换，则退回原位 schema 迁移

解决的问题：

- 代码升级后旧索引触发 `no such column: section_summary`
- 新代码与旧索引之间缺少平滑升级机制

### 2.4 route explanation 收口

落点：

- `backend/tools/tcm_route_tool.py`

已完成：

- 新增 `_normalize_route_reason(...)`
- 将 `origin_entity_forced_hybrid`
- `strategy_graph_override`
  等覆盖逻辑显式归一到统一 route reason 中

解决的问题：

- route 解释串联混乱
- 执行路线已经变更，但 reason 文本不够清晰

### 2.5 图谱证据进一步前置

落点：

- `backend/router/retrieval_strategy.py`
- `backend/services/qa_service/evidence_items.py`
- `backend/services/qa_service/runtime_support.py`
- `backend/services/qa_service/prompts.py`

已完成：

- `answer_policy` 接入响应层
- 图谱证据分组为：
  - `structured`
  - `documentary`
  - `other`
- grounded prompt 中显式区分结构化图谱证据与文献证据
- graph 优先题不再被无意义的 compare 噪声拖回 hybrid

解决的问题：

- 图谱虽已接入，但还不够像“第一公民”
- 关系型问题容易在 route 上被无意义 retrieval 分支稀释

### 2.6 files-first 结构摘要升级框架

落点：

- `backend/services/retrieval_service/files_first_support.py`
- `backend/services/retrieval_service/generate_section_summary_cache.py`
- `backend/services/retrieval_service/engine.py`

已完成：

- `LocalFilesFirstStore` 支持 `summary_cache_path`
- 新增 `SectionSummaryCache`
- 支持通过缓存注入：
  - `section_summary`
  - `topic_tags`
  - `entity_tags`
  - `representative_passages`
- 新增离线脚本：
  - `generate_section_summary_cache.py`

解决的问题：

- 结构摘要原本只有启发式生成
- 现在已经具备升级到离线 LLM 摘要的工程入口

说明：

- 当前运行时默认仍以启发式摘要为主
- 但摘要缓存框架和离线生成入口已经具备

### 2.7 release gate 增强

落点：

- `backend/eval/runners/run_release_gate.py`
- `backend/tests/test_release_gate.py`

已完成：

- Doctoral baseline 统计新增：
  - `deep_fallback_count`
  - `deep_short_answer_count`
- strict gate 不再只看“有没有 answer”
- 现在还会拦截：
  - `planner_deterministic_fallback`
  - Deep 过短答案

解决的问题：

- 旧 gate 会把“极短 fallback 答案”误判为通过
- 现在对 Deep 质量下限更敏感

### 2.8 observability 增强

落点：

- `backend/services/qa_service/runtime_support.py`
- `backend/services/qa_service/helpers.py`

已完成：

- 新增 `generation_diagnostics`
- 记录：
  - prompt 长度
  - 每次生成尝试的 timeout
  - 错误类型
  - fallback 链路
- 最终响应中携带 `generation_diagnostics`

解决的问题：

- 过去只能看到 `deep_llm_fallback:`，很难定位真实根因
- 现在能看到生成层到底是超时、空响应还是 provider 异常

## 3. 直接修复的关键问题

本轮还顺手修复了两个对博士题影响很大的链路问题：

### 3.1 compare entity 噪声污染

症状：

- `q08` 被误抽取为 `气行 / 通阴 / 半夏秫米汤`
- `q10` 被误抽取出大量噪声对象

结果：

- quick 被错误地强制走 `compare_entities_forced_hybrid`
- coverage 长期挂在 `comparison`

修复后：

- 非显式比较题默认不保留 compare entities
- 显式比较题即使 dominant intent 不是 `compare_entities`，也会强制做 compare refinement

### 3.2 graph 优先题无意义 hybrid 化

症状：

- 某些关系型问题明明 graph 证据已够，却仍被 `entity_pair` 拖入 hybrid

修复后：

- strategy 明确要求 graph 优先时，路由层执行 `strategy_graph_override`
- quick 路由更干净，减少不必要 retrieval 分支

## 5. 当前仍然要如实指出的现实

虽然 8 个脆弱点都已做了工程收口，但这不代表系统已经“完全无风险”。

当前仍需正视的现实是：

1. 外部 LLM provider 仍可能出现偶发长尾波动
2. 启发式 section 摘要仍不是最终版语义摘要
3. release gate 虽已更严格，但专家级质量评估仍未完全自动化

也就是说：

> 当前系统已从“脆弱点明显暴露”进入“可控长尾问题”阶段。

## 6. 总结

本轮的价值不在于新增多少模块，而在于把原本零散暴露的 8 个脆弱点尽量收口到工程可控范围：

- Deep 不再一失败就塌成极短答
- coverage 更贴近用户提问维度
- files-first 索引不再对旧 schema 脆弱
- route 解释与真实执行更一致
- graph 证据进一步前置
- section 摘要具备离线升级入口
- gate 能拦截“看似通过、实际很差”的 Deep 答案
- 运行时生成异常终于可诊断

因此，当前系统相较于此前版本，已经从“能跑通高难题”进一步收口为：

> 在高难题、真实服务、可追溯评测条件下更稳定、更可解释、更便于继续优化的问答系统。
