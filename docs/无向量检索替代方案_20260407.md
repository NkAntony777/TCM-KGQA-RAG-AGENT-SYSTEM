# 无向量检索替代方案

更新时间：2026-04-07
适用子项目：`langchain-miniopenclaw/backend`

## 1. 结论

本项目后续不应再以向量库作为主检索链路。更合适的目标架构是：

`files-first + 结构化索引 + skill 编排 + planner 驱动 + 向量兜底可选`

这不是“完全不做索引”，也不是“让 agent 临场翻文件”，而是：

- 文件是事实源
- 结构化索引负责定位
- skill 负责动作
- planner 负责多步决策
- LLM 负责 grounded synthesis
- 向量从主路径降为可关闭的 fallback

## 3. 为什么要替代向量库

### 3.1 当前核心问题不是“不会语义相似”

中医知识问答的主要失真点不是 embedding 不够强，而是：

- 现代问法与古籍旧称错位
- 问题需要书名、篇章、原文片段级别的精确证据
- 相似不等于相关
- chunk 相似命中后，常常缺少可回溯的整节上下文

### 3.2 向量主导会带来的问题

- 召回不可解释
- 命中结果不稳定
- 很难严格控制“只查本书/本章/本实体”
- 对出处题、原文题、比较题支持差
- 评测时难以定位错误来自哪里

## 4. 目标架构

## 4.1 事实源层

- 古籍 txt
- HERB2 转换后的可读证据文件
- 图谱中的 `source_book/source_chapter/source_text`
- 后续可接入现代教材、论文、病例记录

## 4.2 索引层

项目应长期保留四类非向量索引。

### A. 路径索引

- `book://小儿药证直诀/*`
- `chapter://小儿药证直诀/肾虚`
- `entity://六味地黄丸/*`
- `alias://六味地黄丸`
- `case://...`

### B. 实体/别名索引

直接复用 `graph_runtime.db` 中的：

- 实体名
- 别名关系
- 实体类型

主要用途：

- 现代问法归一到古籍名
- 古籍名扩展回现代名
- 比较题、多跳题的实体标准化

### C. 全文索引

主后端使用 SQLite FTS5：

- `book_name`
- `chapter_title`
- `text`
- `filename`
- `file_path`

要求：

- 列权重可调
- 支持 `snippet()` / `highlight()`
- 命中 leaf 后可回 parent section

### D. 关系索引

从图谱运行时库中提取并缓存：

- 实体 -> 书名
- 实体 -> 篇章
- 实体 -> predicate
- 实体 -> 相邻实体

这层不是生成答案，而是约束检索范围。

## 4.3 Skill 层

skill 不负责存储，只负责动作。

建议保留和新增的 skill：

- `expand-entity-alias`
- `locate-book-by-entity`
- `list-book-chapters`
- `read-book-section`
- `grep-book-snippet`
- `trace-origin-passage`
- `read-formula-composition`
- `read-formula-efficacy`
- `compare-formulas`
- `find-case-reference`

每个 skill 必须明确：

- 何时使用
- 何时不用
- 输入
- 输出保证
- 停止条件

## 4.4 Planner 层

deep 模式不再泛化调用 retrieval，而是显式规划 skill。

planner 输出固定结构：

- `gaps`
- `next_actions`
- `stop_reason`
- `source_paths`
- `coverage_after_step`

约束：

- 最多 2 到 3 轮
- 每轮最多 1 到 2 个动作
- 优先补最关键缺口
- 已拿到书名 + 篇章 + 原文片段时停止

## 5. Quick / Deep 新主链

## 5.1 Quick

1. 医疗边界检查
2. query 结构化
3. 实体识别与别名归一
4. files-first 检索
5. 命中后回整节或原段
6. LLM grounded answer

特点：

- 快
- 稳
- 可控
- 默认不走 dense

## 5.2 Deep

1. planner 判断缺口
2. 调 skill 做 2 到 3 轮 follow-up retrieval
3. 每轮更新 evidence bundle
4. 最后做 grounded synthesis
5. 输出 answer + trace + citations

## 6. 替代向量库的分阶段实施

## Phase 1：向量退出主路径

目标：

- quick/deep 默认主链都不走 dense
- files-first 成为默认检索后端

动作：

- `search_mode` 默认切到 `files_first`
- 仅在明确无结果时才允许 fallback
- 前端和 trace 中显式标识是否触发 fallback

## Phase 2：补齐结构化文件索引

目标：

- 让检索从“命中 chunk”升级为“命中书/章/段”

动作：

- 给 classic/HERB2/case 全部补 `book_name/chapter_title/section_key`
- 新增 `list-book-chapters`
- 新增 `chapter://` 全链路展示
- `match_snippet` 进入 citation 生成

## Phase 3：病例库去向量化

当前病例检索仍偏向向量式实现，后续应改为：

- 症状槽位抽取
- 证候标签索引
- 方剂标签索引
- 病例全文 FTS5
- 规则融合排序

排序建议：

- 证候命中优先
- 关键症状重合度次之
- 方剂/治法一致性再次之
- 最后才做全文 lexical 补充

## Phase 4：向量改为可拔插 fallback

目标：

- 默认关闭 dense
- 仅对现代开放问答、跨域弱结构语料开放 fallback

建议：

- 配置项控制是否启用
- trace 中明确记录触发原因
- 评测时将 fallback 命中率作为反向指标

## 7. 评测维度

从现在开始，评测不应再只看“答得像不像”，而要看检索链路质量。

核心指标：

- alias 命中率
- book 命中率
- chapter 命中率
- 原文片段命中率
- 错误书证据率
- fallback 触发率
- 平均响应延迟
- deep 每轮新增有效证据数

## 8. 当前建议的优先级

1. 将 quick/deep 默认主检索切到 `files_first`
2. 补 `list-book-chapters` 和 `read-book-section` skill
3. 将 case 检索从向量改为结构化检索
4. 将 dense fallback 做成可开关配置

## 9. 不建议继续投入的方向

- 继续扩 dense embedding 主链
- 继续把更多问答逻辑绑在 top-k chunk 上
- 让 agent 自由写 SQL/regex/图查询
- 先做完整虚拟 shell 再补检索约束

## 10. 一句话总结

本项目后续不是“优化向量库”，而是把系统正式升级为：

`文件系统式证据层 + 结构化索引 + skill 驱动检索 + planner 控制多步查找`

向量可以保留，但不再作为默认答案链路的核心。
