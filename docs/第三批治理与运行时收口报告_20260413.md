# 第三批治理与运行时收口报告（2026-04-13）

适用子项目：`langchain-miniopenclaw/backend`

## 1. 本轮目标

在第一批、第二批图谱治理完成后，第三批不再继续做大规模物理改边，而是把最终规则真正接进运行时主链，重点解决：

1. 关系族停留在文档中、未进入查询层
2. `source_chapter` 虽然不再是“正文污染”，但仍存在大量 `书名_正文` slug
3. ontology 边界异常边仍会与高质量结构边等权参与查询与路径扩散

## 2. 核心策略

本轮采用的原则是：

- 数据尽量不动
- 查询层先收口
- 限扩散优先于删数据
- 新规则必须有回归测试兜底

也就是说，本轮重点不是“重建一个更干净的库”，而是“让当前库在主链上更安全、更稳定、更可解释”。

### 3.3 normalized predicate / parent 接入

当前已接入的逻辑映射包括：

- `药材基源 -> 拉丁学名`
- `药性特征 -> 归经`

这两类规则当前均以运行时逻辑映射方式生效，而不是再次全量重写图数据库。

### 3.4 source_chapter 规范化

已新增：

- `backend/services/common/evidence_payloads.py`
  - `normalize_source_chapter_label(...)`

当前规则：

- `089-医方论_正文` -> 归一为空 chapter，仅保留 `book://医方论/*`
- `089-医方论_卷上` -> 归一为 `卷上`
- graph / graph_path / syndrome / retrieval / section evidence 共用这套逻辑

这一步的价值在于：

- 不再让历史 `书名_正文` slug 继续污染 evidence path
- 避免 planner 把无意义 slug 误当成精确 chapter scope

### 3.5 ontology 边界治理

本轮没有删除异常边，而是先接入运行时策略：

- 对结构边做 `expected type` 判断
- 结果暴露为 `ontology_boundary_ok`
- ranking 中降权
- path expansion 中限扩散

当前已纳入治理的关系：

- `使用药材`
- `推荐方剂`
- `治疗证候`
- `治疗症状`
- `归经`

1. ontology 异常分层审计
   - 区分“可接受多义边”

2. 治理产物清理策略
   - 历史 `.bak`
   - 临时 `governance*.json/jsonl`

### 7.2 当前不建议做

1. 不建议再次全量重建 SQLite
2. 不建议对 `功效 / 配伍禁忌 / 食忌` 做侵入式拆分
3. 不建议把第三批重新拉回“物理改图优先”

## 8. 一句话总结

第三批当前最重要的成果，不是又改掉了多少边，而是：**图谱治理规则第一次真正进入了在线查询主链。**
