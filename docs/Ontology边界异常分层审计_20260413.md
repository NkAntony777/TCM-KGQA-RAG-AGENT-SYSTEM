# Ontology 边界异常分层审计（2026-04-13）

适用子项目：`langchain-miniopenclaw/backend`

## 1. 本轮目标

上一轮第三批治理已经把 ontology 边界异常接入运行时：

- `ontology_boundary_ok` 标记
- ranking 降权
- path expansion 限扩散

但当时的异常仍是“粗粒度异常池”，还没有区分：

- 可接受多义边

本轮目标就是把这三层真正拆开。

## 4. 关键判断

### 4.1 异常大头不是脏边，而是多义边

最重要的结论是：

- 当前 ontology 异常的大头并不是应该直接删除的脏数据
- 而是“语义仍然成立，但不适合作为结构主边使用”的多义边

典型例子：

- `使用药材`
  - `disease -> herb`
  - `therapy -> herb`
  - `symptom -> herb`
- `推荐方剂`
  - `symptom -> formula`
  - `therapy -> formula`
- `治疗证候`
  - `disease -> syndrome`
  - `symptom -> syndrome`
  - `therapy -> syndrome`
- `治疗症状`
  - `disease -> symptom`
  - `therapy -> symptom`
  - `food -> symptom`
- `归经`
  - `formula -> channel`
  - `food -> channel`

这些边并不一定“错”，但它们不适合与核心结构边等权参与 graph path。

### 4.2 真正优先级最高的是 likely_dirty

相对总量而言，`likely_dirty` 只有 `8936` 条，已经明显小于粗审计时带来的心理压力。

这部分更值得下一阶段优先处理，例如：

- `channel -> herb`
- `book -> symptom`
- `other -> herb`
- `disease -> food`
- `therapy -> food`

这些模式更像抽取错误、引用层污染或边标签误配。

## 5. 分谓词观察

### 5.1 使用药材

- `in-schema = 1459341`
- `acceptable_polysemy = 39216`
- `review_needed = 5930`
- `likely_dirty = 5888`

主要结论：

- `disease / therapy / syndrome / symptom -> herb` 数量很大，但多数仍有信息价值
- `herb -> herb`、`channel -> herb`、`other -> herb` 更可疑

### 5.2 治疗症状

- `in-schema = 583755`
- `acceptable_polysemy = 49019`
- `review_needed = 25291`
- `likely_dirty = 1187`

主要结论：

- 针灸类与治法类文献带来大量 `therapy -> symptom`
- 这类边语义上可以保留，但不应当和方剂/本草直接治疗关系等权
- `book -> symptom` 是当前最明显的脏边候选之一

### 5.3 治疗证候

- `in-schema = 56848`
- `acceptable_polysemy = 6692`
- `review_needed = 1625`
- `likely_dirty = 923`

主要结论：

- `disease -> syndrome` 是最主要的可接受多义边
- `disease -> formula`、`syndrome -> syndrome` 需要人工复核
- `channel -> herb`、`formula -> therapy` 等更像误配

### 5.4 推荐方剂

- `in-schema = 3841`
- `acceptable_polysemy = 610`
- `review_needed = 628`
- `likely_dirty = 55`

主要结论：

- `symptom -> formula` 可视为较自然的多义边
- `formula -> formula` 更像交叉引用或书中串联，建议保留但降权
- `disease -> herb`、`disease -> food` 更接近误配

### 5.5 归经

- `in-schema = 17584`
- `acceptable_polysemy = 499`
- `review_needed = 2126`
- `likely_dirty = 883`

主要结论：

- `formula -> channel` 有一定合理性
- `syndrome / disease / therapy -> channel` 多数应进入人工复核
- `channel -> channel`、`channel -> herb` 更像明显脏边

## 6. 对后续治理的直接建议

### 第一优先级

针对 `likely_dirty` 做小批白名单式清理，不要再对整个异常池动手。

### 第二优先级

把 `review_needed` 按高频组合继续抽样，形成“人工可审”的样本包，再决定是否晋级为可清理规则。

### 第三优先级

`acceptable_polysemy` 保留，不做物理删除；继续走当前运行时策略：

- 降权
- 限扩散
- 不与核心结构边等权桥接

## 7. 一句话结论

第三批下一阶段最重要的判断已经明确：

**ontology 异常的主战场不是“全量清洗”，而是“先清 likely_dirty，再审 review_needed，保留 acceptable_polysemy”。**

补充一句当前状态：

**而且这套分层现在已经进入运行时主链，不再只是离线报告。**
