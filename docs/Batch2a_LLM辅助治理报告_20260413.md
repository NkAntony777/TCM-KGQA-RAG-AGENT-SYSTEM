# Batch2a LLM 辅助治理报告（2026-04-13）

适用子项目：`langchain-miniopenclaw/backend`

## 1. 本轮目标

在 batch1 已经清掉三类最脏的直接删除项后，第二批不再适合继续走纯规则硬删。

原因很明确：

- `使用药材: category -> herb`
 仍然包含分类知识价值
- `使用药材: herb -> herb`
 混有炮制原料、配方列举、可能的配伍信息

因此，本轮目标改为：

- 先用 LLM 做语义判定
- 再只对结论高度一致的子类落库

## 2. 新增能力

### 2.1 LLM 判定脚本

- `backend/scripts/llm_adjudicate_batch2_likely_dirty.py`

特点：

- 复用三元组抽取模块已有的 provider 轮转能力
- 直接使用现有 `call_llm_raw`
- 不另起一套新的 LLM 客户端

输出动作：

- `delete`
- `retype`
- `keep`

### 2.2 小样本验证结论

#### `使用药材: category -> herb`

判定高度一致：

- 稳定输出：`retype -> 属于范畴`

#### `使用药材: herb -> herb`

判定不稳定：

- 有的样本被判 `delete`
- 有的样本被判 `retype -> 配伍禁忌`
- 有的样本被判 `retype -> 属于范畴`

因此：

- `category -> herb` 可以自动治理
- `herb -> herb` 不能直接整批 apply

## 6. 当前阶段判断

现在已经可以明确区分 batch2 的两条分支：

### 可自动推进的分支

- `使用药材: category -> herb`
- 特征：语义稳定、LLM 判定高度一致、适合直接改挂

### 仍需继续拆分的分支

- `使用药材: herb -> herb`
- 特征：语义混杂，不适合整批动作

## 7. 下一步建议

下一轮不该直接继续扩大 batch2a，而应专门拆 `herb -> herb`：

1. 先按 source_text / source_book / 主体模式继续切子类
2. 再分别判断：
   - 炮制/制法原料
   - 配方列举
   - 可能的配伍关系
3. 对每个子类单独决定：
   - delete
   - retype
   - keep

## 8. 一句话结论

batch2 已经不是“能不能用 LLM”的问题了，而是：

**LLM 辅助治理路径已经验证可行，接下来要把 `herb -> herb` 继续拆细，而不是整批硬改。**
