# 意图识别模块适配方案

更新时间：2026-04-04

## 一、背景

外部项目 `E:\TCM_Web_System_v2` 已经有一套相对成熟的意图识别链路，核心价值不在它的疾病标签本身，而在它的工程骨架：

- 实体词典与问题关键词分离
- Aho-Corasick 做多实体命中
- 分类器、解析器、搜索器分层
- 一个问题可以同时命中多个意图候选，再由后续链路决定如何执行

当前项目要吸收的是这套骨架，而不是直接复用旧项目的疾病问答标签。

## 二、哪些东西被吸收了

### 1. 被吸收的部分

- `QuestionClassifier` 的整体分层思路
- 词典驱动的实体识别
- Aho-Corasick 风格的批量实体命中
- 多标签意图候选输出
- “分类结果服务于后续检索策略”这一接口思路

### 2. 没有照搬的部分

- 疾病中心的 `question_types`
- 疾病 / 科室 / 检查 / 药品 / 食物 这套旧标签体系
- 旧项目中的 Neo4j 查询拼装逻辑
- 旧项目里直接把分类结果映射为固定查询模板的做法

原因很明确：当前项目的问答对象是方剂、药材、证候、症状、古籍证据，不是疾病导向问答。

## 三、当前项目里的新模块

当前已新增：

- `backend/router/tcm_intent_classifier.py`

它负责输出统一的 `QueryAnalysis`，包含：

- `matched_entities`
- `entity_types`
- `intent_candidates`
- `dominant_intent`
- `matched_keywords`
- `graph_score`
- `retrieval_score`
- `route_hint`
- `route_reason`
- `graph_query_kind`
- `primary_entity`
- `symptom_name`
- `path_start / path_end`

## 四、当前标签体系

当前已落地的标签不是旧项目的疾病标签，而是围绕本项目检索链路重写：

- `formula_composition`
  组成类问题，只看 `使用药材`
- `formula_efficacy`
  功效类问题，优先看 `功效 / 治法 / 归经`
- `formula_indication`
  主治类问题，优先看 `治疗证候 / 治疗症状 / 治疗疾病 / 推荐方剂`
- `formula_origin`
  出处类问题，优先走 `hybrid`，补图谱、古籍和向量证据
- `syndrome_to_formula`
  从症状或证候出发找方剂或证候链
- `compare_entities`
  比较两个实体，强制走 `hybrid`
- `graph_path`
  显式路径问题
- `open_ended_grounded_qa`
  其他综合型有依据问答

## 五、当前实体识别策略

当前不是依赖单一路径，而是三层兜底：

1. 轻量实体词典
2. `ahocorasick` 自动机
3. 规则抽取兜底

当前轻量词典覆盖：

- 方剂
- 药材
- 证候
- 症状
- 治法
- 古籍书名

如果运行环境里没有 `ahocorasick`，系统会自动回退到子串扫描，不会影响当前链路可用性。

## 六、已经接入到哪里

### 1. `query_router.py`

路由不再独立做一套关键词打分，而是直接消费 `QueryAnalysis`。

这意味着：

- 路由与意图分类共享一套信号
- 不会再出现“路由认为是 retrieval，策略层却按 graph 解释”的漂移

### 2. `retrieval_strategy.py`

策略层现在直接基于 `QueryAnalysis` 生成结构化策略：

- `preferred_route`
- `graph_query_kind`
- `entity_name / symptom_name / path_start / path_end`
- `predicate_allowlist`
- `sources`
- `evidence_paths`
- `candidate_k / final_k`

### 3. `tcm_route_tool.py`

当前路由工具返回中已增加：

- `query_analysis`

这样后续前端、日志、深度模式 Planner 都能直接看到：

- 识别到了哪些实体
- 为什么走 graph / retrieval / hybrid
- 为什么只搜某些谓词

## 七、与快速模式 / 深度模式的关系

### 1. 快速模式

快速模式仍然是工程化固定链路，但现在它的入口已经不再是简单关键词，而是统一分类结果。

作用：

- 更稳地把组成问题压到 `使用药材`
- 更稳地把出处问题压到 `retrieval` 或 `hybrid`
- 更稳地把症状问题压到 `syndrome_chain`

### 2. 深度模式

深度模式后续不应该自己再重新猜一次意图，而应该复用当前 `QueryAnalysis` 作为第一层 Planner 输入。

推荐链路：

1. `QueryAnalysis`
   识别实体、意图、路由倾向
2. `RetrievalStrategy`
   约束本轮检索的谓词、来源和局部 `top_k`
3. `Skill / Tool Execution`
   执行 graph / vector / docs 检索
4. `Evidence Merge`
   统一去重、排序、裁剪上下文
5. `Answer Synthesis`
   只基于证据生成回答

## 八、为什么这比原来的关键词路由更好

原来的问题在于：

- 路由靠一套关键词
- 检索策略靠另一套关键词
- 后续深度模式还可能再来第三套提示词判断

这样会产生三层漂移。

现在的改法把第一步统一成 `QueryAnalysis`，后续所有层都复用它，至少先解决“入口语义不一致”的问题。

## 九、当前边界

这版已经足够支撑当前开发，但仍有边界：

- 实体词典仍然是轻量版，不是全量自动生成
- 没有接入外部 Qwen embedding 问答库做实体补全
- 没有接入完整 skill planner，只是先把结构化策略层打通
- 某些开放式问题仍会落到 `open_ended_grounded_qa`

这不是缺陷，而是当前阶段的合理边界：先把骨架打稳，再扩实体覆盖和深度模式能力。

## 十、后续建议

- 用已发布图谱自动导出实体词表，而不是长期手写词典
- 让深度模式 Planner 直接消费 `query_analysis`
- 把外部问答向量库接入实体对齐和证据路径体系
- 给 `query_analysis` 增加命中置信度或覆盖率指标
- 增加更多真实用户问句回归集，而不是继续堆规则
