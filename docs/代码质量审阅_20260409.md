# 代码质量审阅（2026-04-09）

## 审阅范围

本次审阅聚焦于当前 `langchain-miniopenclaw/backend` 中与 TCM 问答主链收口直接相关的几个点：

- `router/retrieval_strategy.py`
- `router/compare_entity_refiner.py`
- `tools/tcm_service_client.py`
- `services/retrieval_service/hybrid_runtime.py`

本文件是**审阅与文档同步产物**，不直接替代实现说明；实现现状仍以源码和 `docs/PROJECT_PROGRESS.md` 为准。

## 结论摘要

当前代码主方向是正确的：系统已经从“dense 兜底驱动”的旧结构，收口到“graph + files-first + structured index”的默认主链，且 sidecar 不可用时会优先回退到本地真实引擎，这一点显著提升了可用性和可审计性。

同时，当前实现仍处在“主链已切换、兼容层尚未完全删除”的过渡阶段，因此代码质量工作的重点不应是继续扩展功能面，而应是继续压缩兼容分支、把行为边界写清楚，并补足关键退化场景的验证。

## 当前做得好的地方

### 1. 主链与兼容层边界开始清晰化

`retrieval_strategy.py` 已经把“默认主路径”和“兼容来源”明确拆开，文档描述时可以准确区分：

- 已实现的默认主路径
- 仍然保留但不主导的 fallback

这比“看起来什么都支持，但实际默认路径不明确”的状态要健康得多。

### 2. sidecar 故障回退方向正确

`tcm_service_client.py` 当前重点价值不只是“能访问服务”，而是把 sidecar 不可用时的行为收口成：

- graph -> 本地 graph engine
- retrieval -> 本地 retrieval engine

这避免了主链在服务不可达时 silently 回到 mock demo 数据，工程语义明显更可靠。

### 3. files-first 的保护条件更合理

`hybrid_runtime.py` 中对以下情况加了显式保护：

- `vector_compatibility_enabled = false`
- `files_first_dense_fallback_enabled = false`
- `embedding_client` 未就绪
- sparse query 为空

这些 guard 让“为什么没有回退 dense”这件事变得可解释，而不是隐式行为。

### 4. 比较题实体清洗从粗粒度分类器中解耦

`compare_entity_refiner.py` 把比较对象清洗单独抽出来是好方向。它先走启发式，再按需要使用 LLM 精修，既降低了常规路径成本，也给复杂比较题留了兜底空间。

## 当前主要风险

### 1. 仍然存在较多字符串协议

当前若干关键状态依赖字符串约定：

- `backend`
- `retrieval_mode`
- `warning`
- `notes`

这在快速迭代阶段是可接受的，但如果后续前端、评测或 planner 开始基于这些字段做更强依赖，建议逐步把高频状态沉淀为常量或集中枚举，避免同义字符串继续扩散。

### 2. 兼容分支还偏多

`hybrid_runtime.py` 仍承载：

- files-first 主链
- sparse / dense / hybrid fallback
- Milvus / local store 双路径
- rerank / auto merge / lexical gate

这说明它目前仍是一个“过渡期编排中枢”。如果继续往这里叠新行为，后续维护成本会再次上升。建议下一轮优先做的是**删减旧兼容分支**，而不是再扩展新能力。

### 3. LLM 精修路径需要更强的退化验证

`compare_entity_refiner.py` 的降级策略是合理的，但它的正确性目前更多依赖运行时保护而不是文档化约束。后续最好持续验证以下场景：

- 无 API key
- LLM 返回空结构
- LLM 返回少于 2 个比较对象
- 启发式和 LLM 精修结果冲突

### 4. README 曾经落后于当前工程现实

在本次同步前，README 仍偏向旧的通用 Agent / RAG 介绍，对以下事实描述不够明确：

- 当前默认主路径已经不是 vector-first
- backend 依赖管理已经转向 `uv`
- sidecar 故障时应回退本地真实引擎

这类“文档比代码旧一个阶段”的问题，会直接影响项目展示、交接和评测口径。

## 本次文档同步动作

本轮文档同步重点是把 README 更新为和当前工程现实一致的口径：

- 明确默认主链已经切到 `graph + files-first + structured index + skill + planner`
- 明确 dense 仍是兼容层，不是默认主路径
- 明确 backend 推荐使用 `uv sync` / `uv run`
- 明确 sidecar 不可用时应回退到本地真实引擎

## 建议的下一步

1. 继续把“默认主路径”和“兼容 fallback”写成统一术语，避免 README、进展文档、评测脚本各自表述。
2. 为 sidecar 故障、本地 fallback、files-first 禁止 dense 回退等场景维持稳定测试。
3. 下一轮优先减少兼容分支数量，而不是继续把新行为堆进 `hybrid_runtime.py`。

## 一句话结论

当前代码质量的核心判断不是“功能是否还要继续加”，而是：**主链方向已经对了，接下来最值得做的是把过渡兼容层继续收紧，并让文档、测试和运行时行为保持同一口径。**
