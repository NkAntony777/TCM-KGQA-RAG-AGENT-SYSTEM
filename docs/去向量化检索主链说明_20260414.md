# 去向量化检索主链说明

更新时间：2026-04-14  
适用子项目：`langchain-miniopenclaw/backend`

## 1. 结论

本项目当前实现的不是“彻底移除所有向量能力”，而是**将问答主链收口为去向量化检索架构**：

- 默认主链：`graph + files-first + structured index`
- 向量能力定位：兼容层与可关闭 fallback
- 当前目标：优先保证出处可追溯、检索可解释、范围可约束、性能可控

因此，本项目的真实状态应表述为：

> 已完成主链去向量化，保留向量兼容层，但默认不让向量链路主导事实检索。

这一定义比“完全没有向量库”更准确，也更符合当前工程实现。

## 2. 为什么本项目需要去向量化主链

中医知识问答与通用语义检索的核心差异在于，它经常要求：

- 精确到书名、篇名、条文的出处追踪
- 古籍旧称与现代问法之间的实体归一
- 方剂、证候、治法、病机之间的结构化约束
- 比较题、多跳题、鉴别题中的对象消歧
- 命中后能回到整章整节上下文，而不是孤立 chunk

传统 vector-first RAG 在这些点上存在天然缺陷：

- 相似不等于相关
- 很难严格限定“只在某本书/某一章中检索”
- 命中 chunk 后往往缺少稳定的原文上下文
- 评测或调试时不易解释“为什么命中这一段”

因此，本项目没有继续强化“embedding 驱动的全库相似召回”，而是转向：

`结构化路由 -> 逻辑证据路径 -> files-first 精确检索 -> 原文/章节读回 -> grounded synthesis`

## 3. 主链的总体架构

当前主链可以概括为四层：

1. 问题结构化层
2. 非向量索引层
3. 证据路径与 follow-up 层
4. grounded answer 层

其中真正替代向量主链的关键不是单一模块，而是这四层同时成立。

## 4. 问题结构化：先把开放问题变成“可定位问题”

### 4.1 retrieval strategy 先决定“查什么”

系统不会把用户问题直接丢给向量库，而是先经过：

- 意图识别
- 实体识别
- 别名扩展
- 比较对象识别
- 书名偏好识别
- graph / retrieval / hybrid 路由选择

核心入口：

- `backend/router/retrieval_strategy.py`
- `backend/tools/tcm_route_tool.py`

在 `derive_retrieval_strategy(...)` 中，系统会先根据问题类型决定：

- 优先走图谱还是文本
- 谓词允许列表是什么
- 需要哪些 evidence path
- 是否带书名范围
- 是否带病例索引

例如：

- 方剂组成题优先收敛到 `使用药材`
- 功效题优先收敛到 `功效 / 治法 / 归经`
- 出处题优先保留 graph clue + classic docs
- 比较题优先保留 compare entities 与 source paths

### 4.2 逻辑证据路径是主链收口的关键

系统将检索目标组织成逻辑路径，而不是直接组织成“query embedding”：

- `entity://六味地黄丸/功效`
- `entity://小柴胡汤/使用药材`
- `book://金匮要略/*`
- `chapter://伤寒论/辨太阳病脉证并治`
- `alias://小柴胡汤`
- `symptom://胸痹/syndrome_chain`

这一步的意义是：

- 把开放问题转换成“查哪个实体、哪类关系、哪本书、哪一章”
- 让后续检索不需要依赖全库语义相似度猜测
- 为 deep follow-up 提供明确的动作目标

## 5. 四类非向量索引如何协同工作

本项目的“去向量化”并不是不用索引，而是用**更适合中医问答的非向量索引体系**替代 dense 主链。

### 5.1 路径索引

路径索引负责把问题变成可执行入口，路径类型包括：

- `entity://`
- `book://`
- `chapter://`
- `alias://`
- `symptom://`
- `qa://`
- `caseqa://`

对应工具：

- `backend/tools/tcm_evidence_tools.py`

用途：

- 让 planner/quick follow-up 知道接下来应读什么
- 避免 deep 模式退化成无目标的全文搜索
- 为证据补全提供稳定的跳转语义

### 5.2 实体/别名索引

别名索引来自运行时图谱库中的 `别名` 关系，而不是向量近邻。

核心模块：

- `backend/services/qa_service/alias_service.py`

能力包括：

- `aliases_for_entity(...)`
- `detect_entities(...)`
- `expand_query_with_aliases(...)`

作用：

- 将现代问法扩展到古籍旧称
- 让同名异写、别称、简称在 files-first 中也能命中
- 降低“问题表述与古籍文本不一致”带来的漏召回

这一步极其关键，因为中医问答中的大量漏检，并不是语义相似不足，而是**命名系统不统一**。

### 5.3 全文索引：SQLite FTS5 取代 dense 主检索

files-first 文本层的核心不是 Milvus，而是本地 SQLite FTS5。

核心模块：

- `backend/services/retrieval_service/files_first_support.py`

当前实现直接建立：

- `docs` 表
- `docs_fts` FTS5 虚表

索引字段包括：

- `book_name`
- `chapter_title`
- `text`
- `filename`
- `file_path`

查询时使用：

- `MATCH`
- `snippet(...)`
- `bm25(...)`

这意味着当前文本主链是：

1. 先用 tokenized query 命中 leaf chunk
2. 再按 BM25 排序
3. 命中后通过 parent chunk 或章节键回到原段/原节

这与传统 dense RAG 有两个本质区别：

- 命中是可解释的关键词/片段命中，不是隐藏在 embedding 空间中的近邻
- 命中后可以稳定回整节，而不是只拿到一小块切片

### 5.4 结构化 QA / Case 索引

病例和 QA 并不是默认直接走向量库，而是先进入**结构化 SQLite FTS 索引**。

核心模块：

- `backend/services/retrieval_service/qa_structured_store.py`

其中建立了：

- `qa_fts`
- `case_fts`

同样使用：

- `fts5`
- `bm25(...)`

这说明即使是“问答型资料”“病例型资料”，系统默认仍优先使用非向量索引。

只有在结构化索引为空、且显式打开兼容开关时，才会考虑回退到旧向量病例库。

### 5.5 图关系索引

图谱在本项目中不是单纯回答问题，而是承担“范围约束器”和“结构提示器”的角色。

主要作用：

- 实体归一
- 谓词限定
- 关系簇聚合
- 病机链/证候链提示
- 输出 `source_book / source_chapter / source_text`

关键点在于：图谱不会替代文件，但会帮助 files-first 明确“该去哪里读原文”。

## 6. files-first 为什么能够替代向量主检索

### 6.1 文件是事实源，FTS 是定位器

本项目的 files-first 不是“让 agent 手工翻文件”，而是：

- 文件是原始事实源
- FTS 是轻量定位器
- parent chunk / section read 是上下文恢复器

因此它本质上是一种：

> 以本地文档事实源为核心，以全文索引做精确命中，以章节读回保证上下文完整性的检索架构。

### 6.2 chapter:// 与 book:// 使出处追溯成为一等能力

通过共享 evidence path 体系，系统现在可以稳定处理：

- `book://某书/*`
- `chapter://某书/某章`

这意味着：

- 出处题不需要先全库搜索再人工猜书
- deep follow-up 可以直接定向读章节
- 证据回溯能力比 dense chunk retrieval 更强

### 6.3 source scope 让全文检索变成“受控全文检索”

`tcm_route_tool` 在调用 retrieval 时，会传入：

- `search_mode="files_first"`
- `allowed_file_path_prefixes`

这让文本检索不再是“对全库做裸搜”，而是：

- 只在古籍范围搜
- 只在 HERB2 范围搜
- 只在指定来源范围搜

这正是 files-first 真正优于 vector-first 的地方：

> 不是更“语义化”，而是更“受控”、更“可审计”。

## 7. 向量为什么没有消失，但已经退出主链

### 7.1 当前工程状态：兼容保留，默认退居 fallback

当前检索引擎仍保留：

- `MilvusHybridStore`
- `ChromaCaseQAStore`

但这不代表它们仍在主链中占主导。

当前运行时策略是：

- 先走 files-first
- 没结果时先尝试 sparse/local 补救
- 仅当兼容开关开启时，才允许继续掉入 dense fallback

关键配置包括：

- `vector_compatibility_enabled`
- `files_first_dense_fallback_enabled`
- `case_qa_vector_fallback_enabled`

也就是说：

- 向量能力仍在代码里
- 但默认运行策略并不依赖它们回答大多数问题

### 7.2 这是一种工程上更成熟的“去向量化”

完全删除向量代码并不是当前最优选择，因为：

- 兼容旧资产仍有价值
- 某些边缘场景可能仍需 fallback
- 彻底删除会增加未来回滚成本

因此当前更合理的描述是：

> 本项目已经完成主链去向量化，而不是代码级彻底去向量化。

这也是论文和开发报告中更严谨的表述。

## 8. Quick / Deep 模式如何建立在去向量化主链之上

### 8.1 Quick

Quick 的运行顺序已经不是“首轮召回 + LLM 总结”这么简单，而是：

1. route search
2. 获取首轮 graph / retrieval 证据
3. 初始化 coverage state
4. 必要时 `list_evidence_paths`
5. 必要时补 1 到 2 个 follow-up 动作
6. 进入 grounded synthesis

关键点：

- quick follow-up 不会盲目扩大检索范围
- 优先沿已有 evidence path 做最小补证据
- prompt 中默认只使用少量 claim 级证据摘要

### 8.2 Deep

Deep 模式在此基础上进一步显式规划：

- 先列 evidence paths
- 再按 gap 做 follow-up action
- 动作主要是 `read_evidence_path` 与 `search_evidence_text`
- 每轮只补最关键缺口

当前 deep 的能力之所以能够保持质量，并不是因为“搜索更广”，而是因为：

- 它能沿逻辑路径补证据
- 它能回到具体书、章、实体关系
- 它能在 coverage 足够后及时停止

因此，deep 的强项也不是建立在 dense 检索之上，而是建立在：

- evidence path
- files-first
- 结构化 follow-up

之上。

## 9. Prompt 侧也在配合去向量化

当前 grounded prompt 已经明确倾向：

- 优先使用 claim 级摘要
- 不默认复述整段原文
- 只保留有限条数的事实证据

对应效果是：

- 减少长原文 chunk 对 LLM 上下文的浪费
- 避免“把整本书段落灌给模型”的低效做法
- 让生成更聚焦于“证据支持的结论”

因此，本项目不仅在检索层去向量化，也在 prompt 组织层减少了“粗暴拼接 chunk”的旧式 RAG 痕迹。

## 10. 当前主链的实际优势

相较于 vector-first 主链，当前架构的优势主要体现在：

### 10.1 可解释

- 命中来自哪本书、哪一章、哪条关系是明确的
- evidence path 可见
- follow-up 行为可追踪

### 10.2 可约束

- 可以限定古籍、现代文献、病例等来源范围
- 可以限定实体、谓词、章节
- 可以显式绕开不相关来源

### 10.3 可回溯

- 命中后能够回整节
- 可输出 `book:// / chapter:// / entity://` 级证据路径
- 更适合出处型与原文型问题

### 10.4 更适合中医领域

中医知识体系本身就高度依赖：

- 实体名
- 别名
- 书名
- 篇章
- 方证关系
- 原文出处

因此，面向中医领域的高质量问答，天然更适合“结构化约束 + files-first”而不是“全靠语义近邻”。

## 11. 需要避免的误解

### 11.1 误解一：去向量化等于不用索引

错误。  
本项目不是取消索引，而是将索引从“dense 向量索引主导”切换为：

- 路径索引
- 实体/别名索引
- SQLite FTS 索引
- 图关系索引

### 11.2 误解二：files-first 等于直接 grep 文本

错误。  
files-first 不是人工翻文件，而是：

- FTS 精确定位
- parent/section 结构恢复
- 路径与图谱共同约束

### 11.3 误解三：代码里还有 Milvus/Chroma，就不能叫去向量化

不准确。  
是否“去向量化”应看**主链是否依赖向量**，而不是看仓库里是否仍保留兼容模块。

当前状态更准确的说法是：

- 主链去向量化：已完成
- 兼容层向量能力：保留
- 默认主导权：已退出

## 12. 适合在论文/报告中使用的表述

以下表述可直接复用或稍作修改：

### 12.1 一句话版本

本系统采用“graph + files-first + structured index”的主检索架构，以逻辑证据路径和 SQLite FTS5 为核心，完成了问答主链的去向量化收口，向量检索仅作为兼容性 fallback 保留。

### 12.2 方法描述版本

系统首先通过意图识别、实体归一、别名扩展和比较对象抽取，将用户问题转化为可执行的逻辑证据路径，如 `entity://`、`book://`、`chapter://` 等；随后基于本地 SQLite FTS5 files-first 索引进行受控全文检索，并结合图谱关系约束完成章节级原文读回与证据补全；最终由 grounded answer 模块基于 claim 级证据摘要生成答案。该架构避免了传统 vector-first RAG 在出处追溯、范围控制与可解释性方面的不足。

### 12.3 工程优势版本

相较于以向量相似度为主的检索方案，本系统更强调来源可约束、证据可回溯和实体关系可解释，尤其适用于中医经典问答、方证鉴别、条文出处、比较分析等高精度知识任务。

## 13. 后续论文写作时建议强调的创新点

若后续撰写论文或技术报告，建议重点强调以下几点：

1. 将“去向量化”定义为主链收口，而不是模块删除。
2. 强调中医问答与通用 RAG 在证据形态上的差异。
3. 强调 `entity:// / book:// / chapter://` 证据路径体系的设计价值。
4. 强调 SQLite FTS5 + section read 在出处追溯中的优势。
5. 强调图谱在该系统中承担的是“范围约束 + 关系引导”，而非单一答案生成。
6. 强调 quick/deep 都建立在同一套去向量化主链之上，只是证据补全深度不同。

## 15. 最终总结

本项目当前已经形成一条清晰的非向量主链：

`问题结构化 -> 逻辑证据路径 -> graph 约束 -> files-first FTS 命中 -> chapter/book 回读 -> grounded synthesis`

其中，“去向量化”的关键不在于关闭某个库，而在于：

- 用结构化问题理解替代全库语义猜测
- 用路径索引和图谱关系替代无约束召回
- 用 SQLite FTS5 和章节读回替代孤立 chunk 拼接
- 用 claim 级证据摘要替代大段原文灌入

因此，从主链设计、运行时策略和证据组织方式看，本项目已经完成了面向中医知识问答场景的**去向量化检索收口**。
